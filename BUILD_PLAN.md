### Step 1.2 — Supabase Setup with Multi-User Tables
1. Create Supabase project (or use existing)
2. Create `.env.local` from `.env.local.example`
3. Create `supabase/migrations/001_initial.sql` with ALL tables:
   - **auth.users** — Supabase Auth (magic link, email)
   - **users** — User profiles (links to auth.user)
   - **boats** — Boat specs (user_id FK, not singleton)
   - **nav_profiles** — Navigator profiles (experience, crew mode, risk tolerance, etc.)
   - **voyages** — Voyage records (user_id, boat_id, route info, status)
   - **boat_status** — Per-voyage status (voyage_id FK, not singleton)
   - **briefings** — Daily briefings (user_id, voyage_id)
   - **logs** — Logbook entries (user_id, voyage_id)
   - **route_steps** — Per-voyage route legs (generated by AI during onboarding)
   - **checklist** — Per-voyage checklists (voyage_id FK)
   - **chat_history** — Chat messages (user_id, voyage_id)
   - Remove `ai_config` table (no BYOAI)
4. Run migration
5. Generate types: `npx supabase gen types typescript --project-id <id> > src/lib/supabase/types.ts`

## Phase 2: Backend Core (Day 3-4) — DONE

### Step 2.1 — AI Route Generation API
- `src/lib/ai/route.ts` — Route proposal logic:
  - `generateRouteProposals(departure, arrival, boatSpecs, navProfile)` → 2-3 route options
  - `generateCustomRoute(departure, arrival, description, boatSpecs, navProfile)` → 1 custom route
  - Each option: name, summary, distances, estimated days, pros/cons, warnings, detailed steps with coordinates
  - Response is structured JSON (parsed from Claude Sonnet output)
- `src/app/api/ai/route/route.ts` — Route Handler:
  - Auth required (user session)
  - Accepts departure/arrival ports + optional free-text description
  - Reads boat specs + nav profile from Supabase for the user
  - Calls Claude Sonnet via AI proxy
  - Returns structured route options
- Route data is always per-voyage (created during onboarding or from Settings)
- `supabase/seed.sql` contains template data (Audierne→Nice legs, checklist items) as reference/dev aid

### Step 2.2 — Weather API Integration
- `src/lib/weather/open-meteo.ts` — Client for marine + standard weather APIs
  - `getMarineWeather(lat, lon)` → swell, wave height, period
  - `getWeatherForecast(lat, lon)` → wind, gusts, visibility, precip, temp
  - Return typed data, handle errors
- `src/app/api/weather/route.ts` — Proxy route (caches for 1h)

### Step 2.3 — Tides API Integration
- `src/lib/weather/worldtides.ts` — Client for WorldTides API
  - `getTides(lat, lon, days)` → heights, extremes
- `src/app/api/tides/route.ts` — Proxy route (caches for 6h)

### Step 2.4 — AI Proxy (Server-Side)
- `src/lib/ai/proxy.ts` — Core proxy logic:
  - Read `ANTHROPIC_API_KEY` from `process.env`
  - Accept `{messages, model_override?}`
  - Call Claude API with Anthropic SDK
  - Stream response
  - **No ai_config table reading**
  - Get user/voyage context from auth session + database
- `src/app/api/ai/proxy/route.ts` — Route Handler with streaming

### Step 2.5 — Context Builder
- `src/lib/ai/context.ts` — Gathers ALL context for AI prompts:
  - `buildBriefingContext(userId, voyageId)` → full context for daily briefing
  - `buildChatContext(userId, voyageId)` → context for chat messages
  - `buildTriggerContext(userId, voyageId)` → context for trigger evaluation
  - Each pulls: boat status, last logs, route progress, weather, tide, checklist status, active problems, boat specs, nav profile
  - **All queries scoped by user_id and voyage_id**

### Step 2.6 — System Prompts
- `src/lib/ai/prompts.ts`:
  - `BRIEFING_SYSTEM_PROMPT` — Complete briefing prompt with decision profile, format spec
  - `CHAT_SYSTEM_PROMPT` — First mate personality, all context, natural language
  - `TRIGGER_SYSTEM_PROMPT` — Evaluate conditions, generate short messages
  - `ROUTE_SYSTEM_PROMPT` — Generate route proposals from departure/arrival, boat specs, nav profile. Returns structured JSON with 2-3 options (or 1 custom). Considers draft, air draft, fuel range, experience level, risk tolerance.

**Checkpoint:** Can call weather/tides APIs. Can call Claude via AI proxy with multi-user context. Prompts are defined. DONE

## Phase 4: AI Features (Day 6-7) — DONE

### Step 4.1 — Daily Briefing System
- `src/app/api/briefing/route.ts` — Briefing generation endpoint:
  1. Get current user from auth session OR accept voyage_id param for cron
  2. Gather context for voyage (position, weather, tides, boat status, route)
  3. Build prompt with decision profile from nav_profile
  4. Call Claude Sonnet via AI proxy
  5. Parse response (extract verdict, confidence, key data)
  6. Store in `briefings` table with user_id + voyage_id
  7. Send push notification
- `src/app/briefings/page.tsx` — Briefing display:
  - Today's briefing at top (full content)
  - Scrollable history below (filtered by active voyage)
  - Filter by verdict
  - Each briefing shows: date, position, destination, verdict, wind/sea summary
  - **Scoped to active voyage**

### Step 4.2 — AI Chat
- `src/app/api/chat/route.ts` — Chat endpoint:
  1. Get user from auth session
  2. Get active voyage from request
  3. Gather chat context (position, weather, briefing, logs, problems, route)
  4. Load conversation history from `chat_history` (filtered by user_id + voyage_id)
  5. Call Claude Haiku via AI proxy (stream)
  6. Store exchange in `chat_history` with user_id + voyage_id
- `src/app/chat/page.tsx` + `src/components/chat/`:
  - Chat bubbles (user right, AI left)
  - Text input at bottom with send button
  - Suggestion chips ("Today's Briefing", "Fuel Status", "Next Stage")
  - Streaming response display
  - Auto-scroll to bottom on new messages
  - **Scoped to active voyage**

### Step 4.3 — Trigger Engine
- `src/lib/triggers/rules.ts` — 5 trigger rule functions:
  - `checkWeatherChange(context)` → true/false + details
  - `checkLogReminder(context)` → true/false
  - `checkDepartureWatch(context)` → true/false + checklist items
  - `checkCriticalChecklist(context)` → true/false + items
  - `checkLowFuel(context)` → true/false + remaining + next port
- `src/lib/triggers/engine.ts` — Evaluates all rules for a voyage, calls LLM if any fire, sends push
- `src/app/api/ai/triggers/route.ts` — Cron endpoint:
  - Verify CRON_SECRET header
  - **Iterate over all users with active voyages**
  - Evaluate triggers for each
  - Send push notifications per user

### Step 4.4 — Vercel Cron Configuration
Add to `vercel.json`:
```json
{
  "crons": [
    { "path": "/api/briefing", "schedule": "0 4 * * *" },
    { "path": "/api/ai/triggers", "schedule": "0 5,9,13,17,21 * * *" }
  ]
}
```
Note: UTC times (4am UTC = 5am France winter, 6am summer). Adjust for season.

**Cron endpoints must:**
- Query for all users with active voyages
- Loop through and call briefing/trigger logic for each user
- Include proper error handling and logging

**Checkpoint:** Briefing generates and displays. Chat works with streaming. Triggers evaluate and fire. Cron configured for multi-user. DONE

## Environment Variables

```env
# Supabase
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key

# Anthropic AI
ANTHROPIC_API_KEY=your-claude-api-key

# WorldTides
WORLDTIDES_API_KEY=your-api-key

# Web Push (VAPID)
NEXT_PUBLIC_VAPID_PUBLIC_KEY=your-public-key
VAPID_PRIVATE_KEY=your-private-key

# Vercel Cron
CRON_SECRET=your-cron-secret
```

Note: Claude API key is NOT per-user. It's a single shared service key for the entire app (server-side only).

## Key Architectural Differences from BYOAI MVP

- **No ai_config table** — Single ANTHROPIC_API_KEY in env (server-only)
- **User-scoped everything** — All tables have user_id FK, all queries filtered by user_id
- **Voyage context** — Most entities also scoped to voyage_id (boats, logs, briefings, chat, etc.)
- **Cron changes** — Must iterate over all users + voyages, not assume single user
- **Onboarding critical** — Users can't access dashboard until onboarding_completed = true
- **Auth first** — Magic link email provider, ssr library for persistence, protected routes middleware
